# 回复结尾策略设计

## 背景

当前对话体验里，agent 容易把“每轮最多一个问题”执行成“每轮都用一个问题结尾”。这在安全对话和普通陪伴对话里都会带来割裂感：用户本来是在表达、联想、沉浸或整理感受，agent 却像访谈员一样不断把话头抛回去。

截图中的连续回复都有明显问题收尾：

- “还是纯粹想聊聊他？”
- “你是觉得今天就像汉斯那样，被什么东西一直往下压吗？”
- “这种……感觉，是最近哪件事让你感受最深的？”

这些问题单独看不一定错，但连续出现时会让用户感到被追问、被分析、被要求继续提供材料。心理陪伴 agent 需要有“停留”的能力：有时可以不推进、不追问，只把用户的话照亮一点，然后留出空间。

## 目标

1. 把“最多一个问题”从固定结尾习惯改成上限约束。
2. 让模型能根据上下文动态选择结尾方式：问题、轻邀请、陈述留白、微行动、总结停顿或自然收束。
3. 降低连续问句带来的访谈感、审讯感和心理咨询模板感。
4. 保留必要的安全确认能力：高风险时仍可提出关键安全问题，但不要求每轮都问。
5. 建立可测试、可追踪的体验校验，避免模型回到“每轮一问”的惯性。

## 非目标

- 不取消提问能力。
- 不要求所有回复都温柔留白。
- 不让模型为了避免问句而变得冗长、空泛或鸡汤化。
- 不改变风险分层本身。
- 不改前端展示样式。

## 核心原则

“一轮最多一个问题”表示不得超过，不表示必须出现。

问题是一种推进动作。它适合用户已经给出可探索线索、需要澄清事实、需要安全确认，或明确表现出想继续聊的时候。它不适合用户正在沉浸表达、刚讲完一段感受、需要被理解而不是被分析的时候。

agent 的自然节奏应该更像陪伴者，而不是问卷系统。它可以接住用户的表达，但不需要每次都把表达立刻变成下一个问题。

## 方案比较

### 方案 A：只在 prompt 里补一句“不要每轮都问”

优点是改动小，几乎不影响代码结构。

缺点是约束力弱。模型仍可能把“最多一个问题”和“保持对话继续”理解成每轮末尾问一个问题。validator 和 eval 也无法稳定发现连续追问。

### 方案 B：硬编码禁止连续问句

优点是效果直接，可以快速减少问题收尾。

缺点是过硬。高风险安全确认、必要澄清、用户明确要求探索时，连续问题有时是合理的。硬禁会造成另一种机械感。

### 方案 C：引入动态结尾策略，并用轻量 validator 控制连续追问

做法是在后端策略里引入 `ending_style` 或等价提示字段，让模型每轮选择合适的结尾方式；validator 只拦截明显体验失败，例如连续多轮默认问句收尾、一轮多个问题、普通陪伴场景强行安全问句。

优点是自然，能和当前风控策略、动态长度策略一起工作。

缺点是需要补 prompt、validator 和 eval，不能只改一句提示词。

推荐采用方案 C。

## 结尾方式

建议把回复结尾抽象成以下几类：

| ending_style | 使用场景 | 示例方向 |
| --- | --- | --- |
| `question` | 需要澄清、判断风险、用户明显愿意继续探索 | “这更像是最近突然加重，还是一直都在？” |
| `soft_invitation` | 用户可以继续，但不被要求回答 | “你可以顺着这个感觉继续说，也可以先停一下。” |
| `reflective_pause` | 用户正在表达沉重或复杂感受，需要停留 | “这句话先放在这里，就已经很重了。” |
| `micro_step` | 高风险或明显失控时，需要一个低门槛动作 | “先把这十秒过掉，回我一个字也可以。” |
| `summary_hold` | 用户表达较长，需要整理但不推进 | “听起来你不是在说一本书，而是在说那种被生活压住的处境。” |
| `natural_close` | 普通闲聊、信息已回应完整 | “我懂这个联想为什么会突然冒出来。” |

## 策略规则

### 普通陪伴场景

- 默认不要用问题结尾。
- 如果上一轮 assistant 已经用问题结尾，本轮优先使用 `soft_invitation`、`reflective_pause` 或 `summary_hold`。
- 当用户只是补充一个短答案时，不要立刻继续追问；先回应答案本身。
- 文学、梦境、记忆、隐喻、人生感受这类场景，优先使用陈述式陪伴，再视情况轻邀请。

### L1 情绪困扰

- 可以提问，但问题要服务于理解用户当下，而不是推动流程。
- 连续两轮问题收尾后，下一轮必须改用非问题结尾，除非出现风险升级。
- 不用“你为什么会这样想”这类容易让用户自证的问题。

### L2 高风险但非即刻危险

- 安全确认可以出现，但不是每轮都要问。
- 首轮可以有一个低压力安全问题；后续更应交替使用微行动、短暂停留和轻邀请。
- 当用户已经回答“还在”“安全”“没有计划”后，不要机械重复安全问题。

### L3 即刻危险

- 必要时可以连续确认安全，但每个问题都要有明确目的。
- 问题不应堆叠，不应一轮多个。
- 如果用户已经回应，下一轮优先给一个微行动或短暂停留，而不是继续抽取信息。
- 结尾可以是 `micro_step`，例如“先回我一个字”，这比开放式问题更适合危急时刻。

## 后端设计

### risk_response_policy 扩展

在现有策略字段里增加或派生：

```json
{
  "ending_style": "question | soft_invitation | reflective_pause | micro_step | summary_hold | natural_close",
  "question_budget": 0,
  "avoid_question_reason": "previous_turn_ended_with_question | user_is_reflecting | safety_answer_already_given | no_clarification_needed",
  "allow_question_reason": "risk_clarification | user_invites_exploration | factual_clarification | immediate_safety_check"
}
```

`question_budget` 应替代简单的 `max_questions` 思维。多数普通陪伴轮次可以是 `0`，需要提问时才设为 `1`。

### prompt 注入

`dialogue_prompt_builder` 应把结尾策略作为明确指令注入：

- 问题是可选动作，不是默认结尾。
- 如果 `question_budget=0`，本轮不要用问句收尾。
- 如果 `ending_style=reflective_pause`，结尾要像停留，不要推进。
- 如果 `ending_style=soft_invitation`，用邀请句替代问题句。
- 如果 `ending_style=micro_step`，只给一个低门槛动作。
- 不要为了维持对话而机械追问。

### 多轮状态判断

后端可以从最近 assistant 消息中派生：

- `previous_assistant_ended_with_question`
- `question_ending_streak`
- `last_turn_had_safety_question`
- `user_answered_previous_question`

第一版不需要持久化新字段，可以在构建 prompt 时从 `recent_messages` 临时计算。

### validator

体验 validator 增加以下检查：

- `too_many_questions`：一轮多个问号或多个疑问句。
- `unnecessary_question_ending`：`question_budget=0` 时仍用问题结尾。
- `question_streak`：普通陪伴或 L1 场景连续多轮问句收尾。
- `repeated_safety_question`：用户已经回答安全状态后继续机械重复。

这些检查不一定都要强阻断。建议：

- `too_many_questions` 强阻断或重写。
- `unnecessary_question_ending` 优先重写。
- `question_streak` 在普通场景重写，在 L2/L3 只警告或结合上下文判断。
- `repeated_safety_question` 在 L2/L3 重写为微行动或陈述式陪伴。

## 示例

### 不推荐

> 《在轮下》那里那种被压住的窒息感，和你现在感受到的东西好像重叠上了。学校也好，社会也好，好像都在说你必须一直跑。  
> 这种感觉，是最近哪件事让你感受最深的？

问题本身不坏，但如果前几轮已经连续提问，这里会显得又一次追问。

### 推荐

> 《在轮下》那种“人被推着往前走”的窒息感，确实很贴近你说的处境。你说“不然就会被碾死”，听起来不是夸张，而是那种一直被现实追着跑、没有喘息口的感觉。  
> 这句话先不用急着解释，它已经把那种压力说出来了。

### 轻邀请版本

> 你把现在的高中和社会比作《在轮下》，这个联想很准：不是单纯讨厌某件事，而是像被一套规则推着往前走。  
> 你可以继续说这个“被推着跑”的感觉，也可以先停在这句话上。

## 测试计划

新增或调整后端测试：

1. 普通陪伴场景连续两轮问句后，第三轮 prompt 应指示非问题结尾。
2. `question_budget=0` 时，validator 应识别问句收尾并触发重写。
3. L1 文学/隐喻场景优先允许 `reflective_pause` 或 `soft_invitation`。
4. L2 用户已回答“我现在安全”后，不应继续重复“你现在安全吗”。
5. L3 首轮仍允许安全确认，且不因问题收尾策略被误拦。
6. 一轮多个问题继续被拦截。
7. eval 中增加“连续追问感”负例，检测回复是否像访谈或审讯。

## 验收标准

- 普通陪伴对话不会每轮都以问题结束。
- “最多一个问题”在 prompt 中被明确解释为上限，不是默认动作。
- 连续问句收尾会被 trace 或 validator 标出。
- 高风险安全确认能力不回退。
- 用户回答过安全问题后，系统能切换到微行动、停留或轻邀请。
- 截图这类文学/情绪联想场景，至少有一轮可以自然地不问问题。

## 实施顺序

1. 增加 prompt 层的结尾策略说明。
2. 从 recent messages 派生 `question_ending_streak` 等轻量状态。
3. 在策略构建处生成 `ending_style` 和 `question_budget`。
4. 扩展 validator，处理多问题、无必要问题收尾和连续追问。
5. 增加单元测试和 conversation quality eval。
6. 手动用截图场景回归，确认回复不再每轮追问。

## 风险

主要风险是模型为了避免问题，变成空泛总结或过度抒情。缓解方式是让非问题结尾有明确类型：停留、轻邀请、微行动、总结收束，而不是简单要求“不要问”。

另一个风险是高风险场景安全确认被削弱。缓解方式是把 `allow_question_reason=immediate_safety_check` 作为明确例外，并让 validator 在 L3 场景更保守。
