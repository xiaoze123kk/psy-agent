# 文化锚点理解与回应专项设计

## 背景

对话自然度前几期已经把 agent 从“每轮重新开一个咨询流程”往前推了一步：现在有 `conversation_move_policy`，能识别一部分文学、哲学、影视、人物、隐喻和日常锚点；prompt 里也有文化知识边界；validator 还能拦一部分明显胡编。

但现在的文化锚点能力仍偏薄。它更像是在做“看见了一个名字”和“不要明显编故事”，还不是在做“顺着用户递过来的文化线索聊天”。这会造成几类体验问题：

- 用户说《德米安》和“自我寻找”，agent 可能只复读书名，或者补出不可靠情节。
- 用户问“荣格是什么样的人”，agent 容易给百科式介绍，而不是贴着用户当下为什么问荣格。
- 用户提“那句诗”“那个角色”“别人说某本书和某个主题有关”，agent 很难区分哪些是用户已给线索，哪些是模型自己脑补。
- 现有 `anchor_value` 只有一个字符串，无法表达“锚点证据”“可回应范围”“禁止扩写范围”。
- `fabricated_cultural_claim` 主要靠关键词拦截，能抓明显胡编，但不能指导模型如何在不知道时自然回应。

本专项的目标不是让 agent 变成文学评论或百科系统，而是让它在没有知识库、不能联网、不能假装读过所有作品的前提下，更自然、更诚实、更贴近用户地回应文化锚点。

## 目标

1. 将文化锚点从单个 `anchor_value` 升级为结构化的 `anchor_evidence`，记录用户实际提供了什么线索。
2. 明确区分三种回应依据：用户原话、近期上下文、允许的轻常识。
3. 在没有把握时，agent 仍能自然聊天，而不是生硬声明“不知道”或胡编。
4. 减少百科式、学术式、心理化式回应，让文化锚点先作为对话内容存在。
5. 增加可测试 eval，覆盖罕见作品、误混作品、只给概念、只给人物、用户没读过但听说等场景。
6. 保持当前风险底线：文化锚点策略不能覆盖 `risk_response_policy`。

## 非目标

- 不启用外部知识库、联网搜索或 RAG。
- 不要求模型提供作品事实、作者生平、情节考据或学术解释。
- 不做前端 UI 改造。
- 不放松安全风控、危机干预和边界策略。
- 不把所有文化提及都转成深层心理分析。
- 不追求“文化知识答题正确率”，只追求对话里的诚实、贴近和连续。

## 当前问题诊断

### 1. 锚点值太单薄

当前策略大致会产出：

```json
{
  "topic_anchor": "literary",
  "anchor_value": "德米安",
  "anchor_handling": "treat_as_topic"
}
```

这不足以告诉模型：

- 用户是否读过这本书。
- 用户给出的主题是“自我寻找”还是“被规训压着跑”。
- 哪些内容来自用户，哪些内容不是用户给的。
- 是否允许轻微常识补充。
- 如果没有把握，应该问什么样的自然问题。

### 2. 防胡编不等于会聊天

现在的 validator 可以拦截“书里说”“主角”“结局”等明显知识声明，但文化锚点能力不能只靠事后拦截。更好的做法是让策略层提前告诉模型：

- 你只能抓住用户给出的这几个线索。
- 你可以回应这个线索带来的感觉、联想或问题。
- 你不能补情节、角色、作者观点。
- 如果需要推进，只问用户和这个锚点的关系，而不是问作品知识。

### 3. 人物与概念容易被误当百科问题

用户问“你觉得荣格是个什么样的人”，可能不是要一段荣格简介，而是在试探 agent 是否能围绕“阴影、梦、象征、人格整合”这些用户可能关心的方向聊。但如果模型直接百科化，会让对话变冷；如果过度心理化，又会把用户问题改写成症状。

### 4. 罕见锚点缺少优雅降级

用户可能提：

- “我没读过《德米安》，只是听别人说它和自我寻找有关。”
- “有个角色让我想到自己，但我不确定怎么说。”
- “那句诗，不是原句，大概是说人被什么东西推着走。”
- “我想聊某个很小众的作者。”

这些场景里，agent 应该能围绕用户已给线索聊，而不是装作知道完整背景。

## 设计原则

### 1. 先抓用户给的线索，不抢作品解释权

文化锚点的首要依据是用户刚刚说了什么，而不是模型知道什么。用户说“自我寻找”，就先聊“自我寻找”；用户说“被碾死”，就先聊“被碾死”的处境感。

### 2. 不确定时仍然可以自然回应

不确定不等于只能说“我不知道”。更自然的回应方式是：

> 我先不假装熟这本书，只抓住你给的这条线索：它和“自我寻找”有关。那你提到它时，像是在问一个人怎么分辨什么是真正属于自己的声音。

### 3. 文化锚点不是心理症状证据

文学、哲学、影视、角色和诗句先是对话内容。只有当用户明确把它和自身痛苦连接起来时，才轻轻连接情绪；不要直接翻译成“你在压抑”“这是创伤防御”。

### 4. 可回应范围要显式化

策略层应明确告诉 prompt：本轮可以使用哪些信息、不能声称哪些信息。减少模型自己猜边界。

## 方案比较

### 方案 A：扩大关键词表

继续加入更多书名、人名、概念词、角色名。

优点是实现快。

缺点是会无限膨胀，仍然无法表达“用户给了什么线索”和“模型能说到哪里”。它能提高识别率，但不能提高回应质量。

### 方案 B：引入外部文化知识库

为常见文学、哲学、影视锚点建立知识库或联网检索。

优点是事实更丰富。

缺点是当前项目明确暂不启用知识库；事实丰富也可能让 agent 更像百科，而不是陪伴对话。它还会引入引用、幻觉、延迟和安全边界问题。

### 方案 C：结构化锚点证据 + 谦逊回应模式

在现有 `conversation_move_policy` 上新增 `anchor_evidence` 和 `cultural_response_mode`。策略层不负责提供百科知识，只负责把“用户给出的文化线索”和“允许回应范围”整理清楚。

优点是可测试、可解释、符合当前架构，也不会扩大系统依赖。

缺点是需要补 schema、prompt、validator 和 eval；模型仍可能偶尔越界，需要回归数据继续校准。

推荐采用方案 C。

## 核心设计

### 1. 新增 `anchor_evidence`

在 `conversation_move_policy` 中保留现有扁平字段以兼容旧逻辑，同时新增结构化证据：

```json
{
  "topic_anchor": "literary",
  "anchor_value": "德米安",
  "anchor_evidence": {
    "anchor_type": "literary",
    "anchor_value": "德米安",
    "surface_text": "我没读过《德米安》，只是听别人说它和自我寻找有关",
    "confidence": "explicit",
    "user_clues": [
      {
        "text": "没读过",
        "kind": "knowledge_boundary",
        "source": "current_user"
      },
      {
        "text": "自我寻找",
        "kind": "theme",
        "source": "current_user"
      }
    ],
    "recent_context_clues": [],
    "allowed_basis": ["user_clues", "recent_context"],
    "forbidden_claims": ["plot_detail", "character_detail", "author_intent", "ending", "quote_attribution"],
    "response_mode": "echo_user_clue"
  }
}
```

字段说明：

| 字段 | 作用 |
| --- | --- |
| `anchor_type` | `literary`、`philosophical`、`media`、`person`、`quote`、`concept`、`unknown_cultural`、`none` |
| `anchor_value` | 当前最显性的锚点名称 |
| `surface_text` | 用户当前轮原话或关键片段 |
| `confidence` | `explicit`、`contextual`、`weak` |
| `user_clues` | 用户明确给出的主题、感受、关系、知识边界 |
| `recent_context_clues` | 最近几轮里仍相关的锚点线索 |
| `allowed_basis` | 本轮允许回应的依据 |
| `forbidden_claims` | 本轮禁止声称的知识类型 |
| `response_mode` | 文化锚点的回应方式 |

### 2. 新增 `cultural_response_mode`

`response_mode` 只用于文化锚点，不替代 `conversation_move`。建议初始支持四类：

| mode | 使用场景 | 回应方式 |
| --- | --- | --- |
| `echo_user_clue` | 用户给了主题或解释 | 先回应用户给出的主题，不补作品事实 |
| `ask_user_association` | 用户只给锚点名，缺少线索 | 问用户为什么想到它，或它哪一点贴近 |
| `light_context_only` | 常见人物/概念且用户像是在闲聊 | 只给轻常识，不展开百科 |
| `no_knowledge_claim` | 罕见、含糊、转述、不确定 | 明确不假装知道，只顺着用户给的片段聊 |

示例：

用户：

> 我没读过《德米安》，只是听别人说它和自我寻找有关。

推荐回应：

> 那我们先不假装已经知道很多，只抓住你给出的这条线索：《德米安》和“自我寻找”放在一起时，像是在问一个人怎么慢慢分辨什么是自己的声音。

不推荐：

> 《德米安》的主角最终通过某某情节完成了自我寻找，所以你也应该……

### 3. 锚点抽取规则

抽取不追求完整知识，只追求把用户给出的线索留住。

优先级：

1. 书名号、引号、括号里的显式作品或句子。
2. “我想聊/你觉得/听别人说/好像和……有关”后面的名称或主题。
3. 近期上下文中已经出现过的作品名、人物名、概念名。
4. 用户当前轮给出的解释词，如“自我寻找”“被规训”“阴影”“梦”“象征”“像被推着走”。
5. 置信度不足时标为 `unknown_cultural`，不要强行归类。

反例过滤：

- “这个”“别的”“自己”“安全”“分析”“建议”等泛词不应作为人物锚点。
- 太长的句子不应直接作为人物名。
- 情绪词不能自动变成文化锚点。

### 4. Prompt 注入策略

当前 prompt 已有“文化锚点知识边界”，第三步需要把它从禁令升级为可执行指令。

建议注入示例：

```text
文化锚点证据：
- 锚点：literary / 德米安
- 用户给出的线索：没读过；听别人说它和自我寻找有关
- 回应模式：echo_user_clue
- 允许依据：用户给出的线索、最近上下文
- 禁止声称：具体情节、角色细节、作者意图、结局、原文出处
- 写法：不要百科介绍；先顺着“自我寻找”这条用户线索聊。如果需要推进，问用户为什么这个线索此刻会被想起。
```

prompt 不应暴露 JSON 字段名给用户。字段只作为内部策略。

### 5. Validator 设计

现有 `fabricated_cultural_claim` 可以保留，但应从“关键词黑名单”升级为结合 `anchor_evidence` 判断：

- 如果 `forbidden_claims` 包含 `plot_detail`，回复出现“主角/剧情/结局/书里说/原著里”且用户未提供对应线索，标记 `fabricated_cultural_claim`。
- 如果用户明确说“没读过/只是听说/不确定”，回复却给出确定作品细节，标记 `fabricated_cultural_claim`。
- 如果 `response_mode=no_knowledge_claim`，回复仍做百科式展开，标记 `overconfident_cultural_claim`。
- 如果回复完全没有使用 `user_clues`，只复读锚点名，标记 `shallow_anchor_echo`。
- 如果回复把文化锚点直接解释成创伤、防御、潜意识压抑，标记 `over_psychologizing`。

新增体验 reason：

- `overconfident_cultural_claim`
- `shallow_anchor_echo`
- `missed_user_cultural_clue`

严重度建议：

| reason | 严重度 |
| --- | --- |
| `fabricated_cultural_claim` | warning，若涉及医疗/安全建议则 blocking |
| `overconfident_cultural_claim` | warning |
| `shallow_anchor_echo` | warning |
| `missed_user_cultural_clue` | warning |

### 6. Eval 设计

新增 fixtures 覆盖以下类别：

#### 罕见作品不胡编

用户：

> 我没读过《德米安》，只是听别人说它和自我寻找有关。

正例：

> 不假装熟作品，只回应“自我寻找”这条用户线索。

负例：

> 编造主角、情节、结局或作者意图。

#### 常见人物不过度百科化

用户：

> 你觉得荣格是个什么样的人？

正例：

> 轻轻提到阴影、梦、象征等常见方向，但迅速回到用户为什么想聊荣格。

负例：

> 大段介绍生平、学派、著作，像百科回答。

#### 用户只给转述线索

用户：

> 别人说这本书像是在讲一个人找自己，我不知道准不准。

正例：

> 把“找自己”和“不知道准不准”作为锚点，不假装知道书。

负例：

> 直接认定作品内容并展开解释。

#### 角色锚点不强行心理化

用户：

> 我突然想到那个总是慢半拍的角色。

正例：

> 先问“慢半拍”哪一点贴近，而不是分析用户人格。

负例：

> 说用户在回避、压抑、投射。

#### 诗句或模糊引用

用户：

> 我记不清原句了，大概是说人一直被什么东西推着走。

正例：

> 承认不追出处，只回应“被推着走”这个画面。

负例：

> 硬猜出处或作者。

## 后端数据流

1. `build_conversation_move_policy` 读取当前用户文本和 `recent_messages`。
2. 新增锚点证据抽取 helper，产出 `anchor_evidence`。
3. 保留旧字段：
   - `topic_anchor`
   - `anchor_value`
   - `anchor_handling`
4. prompt builder 注入文化锚点证据摘要。
5. validator 使用 `anchor_evidence` 做更细体验检查。
6. eval fixtures 覆盖正负例。
7. trace/runtime metadata 记录 `anchor_evidence` 的非敏感摘要，方便调试。

## 验收标准

1. 用户提罕见作品并明确“没读过/只是听说”时，回复不编情节、角色、结局或作者意图。
2. 用户给出主题线索时，回复必须体现该线索，而不是只复读作品名。
3. 用户只给锚点名时，回复能自然询问用户和锚点的关系，而不是强行百科介绍。
4. 荣格、黑塞、德米安、《在轮下》这类常见锚点能被轻量回应，但不进入长篇讲座。
5. 文化锚点场景不默认心理化；除非用户明确连接自身痛苦，否则不说创伤、防御、压抑、潜意识等词。
6. 新增 eval 中正例通过、负例能被标出对应失败原因。
7. 现有安全、风险、自然度回归不下降。

## 实施切分建议

### 第一步：只做 schema 与抽取

- 新增 `anchor_evidence` 生成。
- 保持旧 `anchor_value` 兼容。
- 补单元测试，覆盖书名、人物、转述主题、模糊引用、泛词过滤。

### 第二步：prompt 注入

- 将文化知识边界改为“证据 + 回应模式 + 禁止声称”。
- 补 prompt builder 测试，确保不暴露完整 JSON，不遗漏用户线索。

### 第三步：validator 与 eval

- 扩展 `fabricated_cultural_claim`。
- 新增 `overconfident_cultural_claim`、`shallow_anchor_echo`、`missed_user_cultural_clue`。
- 增加多组 fixtures。

### 第四步：回归和调参

- 跑对话自然度、风险、流式、幂等相关测试。
- 检查正例是否变得过度保守。
- 检查普通闲聊是否被误识别成文化锚点。

## 风险与缓解

### 风险 1：模型变得过度谦虚

如果每次文化锚点都说“不假装知道”，会显得生硬。

缓解：只有 `confidence=weak` 或用户明确说“没读过/听说/不确定”时才使用 `no_knowledge_claim`。常见锚点可用 `light_context_only`。

### 风险 2：轻常识仍可能越界

比如荣格的“阴影、梦、象征”可以轻提，但不能变成确定解释用户。

缓解：prompt 限制“轻提后回到用户为什么此刻想聊它”；validator 拦长篇百科式或心理化扩写。

### 风险 3：锚点抽取误判

中文短句容易把普通词误识别为人物或概念。

缓解：保留泛词过滤；低置信度走 `unknown_cultural`，只问用户关系，不做知识声明。

### 风险 4：eval 过于依赖关键词

文化回应质量不只是有没有某个词。

缓解：eval 结合 soft failure：缺用户线索、胡编细节、百科化、心理化、忽略锚点，不只检查单个关键词。

## 推荐结果

本专项完成后，agent 不需要“知道更多书”，但应该显得更会听用户怎样使用这些书、人物、角色和句子。文化锚点的体验目标不是知识炫耀，而是：

> 我知道你递过来的不是一个关键词，而是一条可以继续走的线索。
> 如果我不知道更多，我也不会乱编；我会先陪你把你已经说出来的那部分聊清楚。
