# 前端单一空白对话与历史对话治理设计

## 背景

当前前端左侧“开始新对话”会立即调用 `POST /api/v1/chat/threads` 创建后端线程，并把返回的线程插入左侧列表。用户连续点击按钮时，即使没有输入任何消息，也会产生多条标题相同、摘要为空、风险级别为 `L0` 的“新的陪伴对话”。截图中的左侧列表已经出现多条空对话，降低了续聊入口的可读性，也会污染后端线程数据。

当前相关代码集中在：

- `frontend/src/app/ningyu/NingyuAppShell.tsx`
  - `handleStartNewThread()` 立即调用 `api.startThread()`。
  - `activateThread()` 把新线程插入 `threads` 列表并清空消息。
  - `homeEntries` 直接把 `threads` 映射到左侧列表。
- `frontend/src/api/endpoints.ts`
  - `startThread()`、`listThreads()`、`listMessages()` 已存在。
- `frontend/src/types/api.ts`
  - `ThreadListItem` 目前只有 `thread_id`、`title`、`mode`、`last_summary`、`last_risk_level`、`updated_at`。
- `backend/app/api/v1/endpoints/chat.py`
  - `GET /threads` 当前不支持分页、搜索、归档筛选，也不返回消息数量或 `is_empty`。

## 用户问题

1. 左侧“新对话”太多，应该只能有一个新的空对话。
2. 如果大量对话堆积，需要一个不会把侧栏撑爆的产品方案。
3. 设计要符合前端产品体验，不只是把重复项隐藏掉。

## 设计目标

1. “开始新对话”在用户尚未发送第一条消息前，不应反复创建后端线程。
2. 任意时刻前端最多展示一个空白新对话入口。
3. 用户已经开始过的真实对话不能被误删、误隐藏或误归档。
4. 大量历史对话堆积时，左侧栏保持可扫描、可续聊、低打扰。
5. 方案能先以前端改动止血，再为后端分页、归档、空线程清理留出接口演进路径。

## 非目标

- 不在本轮 spec 中实现代码。
- 不重做整个 Ningyu 首页或聊天工作台视觉系统。
- 不默认删除历史对话数据。
- 不把心理咨询会话做成强运营式列表管理台；侧栏仍以“安静续聊”为主。
- 不编辑 `frontend/dist/`、`node_modules/`、缓存或日志文件。

## 设计原则

### 1. 空白对话是草稿，不是历史

空白对话在用户输入第一条消息前只是“当前准备开始的聊天草稿”。它不应占用后端线程，也不应进入历史列表。只有用户真正发送消息后，才创建后端线程并进入历史。

### 2. 历史对话是记忆入口，不是无限流水账

心理咨询场景里的历史对话可能承载连续性和安全信息，不能粗暴删除。大量对话堆积时，应优先提供分组、搜索、归档和加载更多，而不是让所有项目挤在一个滚动列表里。

### 3. 侧栏服务于恢复上下文

左侧栏优先展示“当前草稿”“最近真实对话”“需要留意的高风险或近期对话”。更早的低风险对话可以折叠或进入归档视图。

## 方案选项

### 方案 A：前端复用已有空线程

点击“开始新对话”时，如果当前列表里已经有一个看起来为空的线程，就只激活它，不再调用 `startThread()`。

优点：

- 改动小。
- 不需要修改发送流程。
- 能立即减少继续点击造成的新空线程。

缺点：

- 已经创建的空线程仍会留在后端。
- 当前 `ThreadListItem` 没有 `message_count` 或 `is_empty`，只能用标题、摘要、风险级别做启发式判断，可能误判。
- 仍然把“空白草稿”建成后端历史，概念不干净。

### 方案 B：本地草稿优先，首条消息时再建线程

点击“开始新对话”只创建一个本地 `draft` 状态，不调用后端。用户发送第一条消息时，前端先调用 `startThread()`，成功后再向新线程发送消息流。若再次点击“开始新对话”，只聚焦已有草稿。

优点：

- 从源头避免空线程污染。
- “只能有一个新的空对话”可以由前端状态保证。
- 用户体验自然：先看到空白聊天区，真正输入后才进入历史。

缺点：

- 发送流程需要支持“无 `activeThreadId` 但存在本地草稿”的状态。
- 首条消息会多一次创建线程请求，错误态需要设计清楚。
- 需要小心处理快速连点和流式发送中的禁用状态。

### 方案 C：后端提供草稿线程与归档治理接口

后端增加“获取或创建唯一草稿线程”、`message_count`、`is_empty`、分页、搜索、归档、删除空线程等接口，由后端保证唯一性和清理。

优点：

- 数据一致性最好。
- 可彻底治理历史堆积。
- 多端登录也能共享“唯一草稿”规则。

缺点：

- 跨前后端改动更大。
- 需要数据库迁移或至少接口契约调整。
- 不适合作为立即止血的第一步。

## 推荐方案

采用“方案 B 为主，方案 C 渐进补强”的两阶段设计。

第一阶段只改前端：把“新对话”改成本地单例草稿，发送第一条消息时再创建后端线程。同时对历史列表做前端分组和重复空线程折叠，先解决截图中的重复空对话体验。

第二阶段再补后端接口：线程列表返回 `message_count`、`is_empty`、分页游标和归档状态，并提供空线程清理或归档能力。这样可以治理已有脏数据和未来多端场景。

## 前端交互设计

### 新对话按钮

- 默认文案仍为“开始新对话”。
- 点击后：
  - 如果当前已有本地草稿，激活该草稿并清空右侧消息区。
  - 如果没有草稿，创建一个本地草稿，标题显示“新的陪伴对话”。
  - 不调用 `api.startThread()`。
- 当草稿处于激活状态且没有消息时：
  - 按钮可以显示为“继续当前新对话”或保持“开始新对话”，但行为必须是聚焦当前草稿。
  - 按钮不应新增第二个草稿。
- 当 `chatStreamStatus === "streaming"` 或首条消息正在创建线程时，按钮禁用，避免产生竞态。

### 草稿在左侧列表中的呈现

草稿固定显示在左侧列表顶部，样式应比历史对话更轻：

- 标题：“新的陪伴对话”。
- 摘要：“还没有开始，写下第一句话后会保存。”
- 时间：“草稿”。
- 不显示 `L0` 或 `companion` 这类技术标签。
- 只能有一个草稿项。

如果用户切换到历史对话，草稿可以保留在顶部；再次点击它回到空白聊天区。如果用户刷新页面，草稿不恢复，因为它没有真实内容。

### 首条消息发送

当用户在草稿状态发送消息：

1. 前端进入 `creatingThreadForDraft` 状态。
2. 调用 `api.startThread({ mode: "companion", title: "新的陪伴对话" })`。
3. 成功后用返回的 `thread_id` 替换草稿状态，调用现有流式发送逻辑。
4. 新线程进入历史列表顶部，草稿消失。
5. 如果创建线程失败：
   - 保留草稿和用户输入。
   - 显示错误：“新对话暂时没创建成功，可以稍后重试。”
   - 不生成后端线程，不丢用户草稿。

### 旧空线程折叠

第一阶段不删除任何历史线程，但应减少视觉噪声：

- 只把最新一个“疑似空线程”显示在列表中。
- 旧的疑似空线程默认折叠到“隐藏的空白对话”计数，不作为普通续聊项展示。
- 疑似空线程的第一阶段判断只用于视觉折叠：
  - `last_summary === null`
  - `last_risk_level === "L0"`
  - `title` 为“新的陪伴对话”“未命名对话”或后端默认的 `new session`
- 不符合这些条件的线程不能被折叠。
- 第二阶段由后端 `message_count === 0` 或 `is_empty === true` 替代启发式判断。

## 大量对话堆积处理

### 左侧默认展示策略

侧栏不应无限展示所有线程。推荐结构：

1. 顶部固定区：当前草稿或当前激活对话。
2. 最近对话：显示最近 10 到 12 条真实对话。
3. 时间分组：
   - 今天
   - 近 7 天
   - 更早
4. “更早”默认折叠，只显示入口和数量。
5. 超过默认数量时显示“查看更多”，进入历史抽屉或历史页。

这样可以让左侧保持陪伴工具的安静感，而不是变成拥挤的数据库列表。

### 搜索与筛选

当真实线程数超过 20 条时，左侧应出现轻量搜索入口：

- 搜索标题和摘要。
- 可筛选“有风险提醒”“有摘要”“最近 7 天”。
- 搜索结果应替换列表区域，不打开新的复杂页面。
- 搜索框在移动端可折叠为图标按钮，避免挤占聊天空间。

### 归档而不是删除

历史对话默认提供“归档”而不是“删除”：

- 普通用户入口使用“收起到归档”文案，降低误删压力。
- 归档线程不出现在默认左侧列表。
- 归档视图可以通过“历史对话”或“查看更多”进入。
- 删除只用于明确的空白线程清理或未来设置页中的高级操作。

### 置顶与风险优先

后续可支持：

- 置顶对话固定在最近列表上方。
- `L2` / `L3` 相关线程不被自动隐藏到“更早”，除非用户手动归档。
- 高风险线程显示更清楚的安全标记，但不要用强烈刺激的视觉样式。

## 数据与状态模型

前端需要引入本地草稿状态：

```ts
interface DraftThread {
  id: "local-draft";
  title: "新的陪伴对话";
  createdAt: string;
}
```

核心状态关系：

- `activeThreadId: string | null` 继续代表真实后端线程。
- 新增 `activeDraftThread: DraftThread | null` 表示本地草稿。
- 当草稿激活时，`activeThreadId` 为 `null`，`messages` 为空。
- 当用户选择真实线程时，`activeDraftThread` 可以保留，但不激活。
- 当草稿首条消息成功创建真实线程后，清空 `activeDraftThread`。

第二阶段后端可扩展：

```ts
interface ThreadListItem {
  thread_id: string;
  title: string;
  mode: string;
  last_summary: string | null;
  last_risk_level: "L0" | "L1" | "L2" | "L3";
  updated_at: string;
  message_count: number;
  is_empty: boolean;
  archived_at: string | null;
}
```

列表接口可扩展为：

- `GET /api/v1/chat/threads?limit=20&cursor=...&archived=false&q=...`
- `PATCH /api/v1/chat/threads/{thread_id}/archive`
- `DELETE /api/v1/chat/threads/{thread_id}` 仅用于空线程或明确确认后的删除。

## 组件边界

第一阶段建议保持文件改动克制：

- `frontend/src/app/ningyu/NingyuAppShell.tsx`
  - 增加草稿状态。
  - 调整 `handleStartNewThread()`。
  - 调整 `handleSend()` 支持草稿首发。
  - 增加线程列表归一化与分组逻辑。
- 可选新增 `frontend/src/app/ningyu/threadList.ts`
  - 放置 `buildThreadEntries()`、`isProbablyEmptyThread()`、`groupThreadEntries()`。
  - 避免 `NingyuAppShell.tsx` 继续膨胀。
- 可选新增 `frontend/src/app/ningyu/threadList.contract.ts`
  - 用 TypeScript 编译期断言或小型无框架测试覆盖列表折叠规则。

## 视觉与可访问性

设计方向应是“安静、可扫描、低干扰”的陪伴工作台：

- 草稿项视觉更轻，不使用和真实历史完全相同的强边框高亮。
- 当前激活项保留清晰选中态。
- 大量历史使用分组标题和折叠，不靠更细的滚动条解决。
- 按钮、列表项、搜索入口需要键盘可达。
- 列表项标题和摘要必须单行或两行截断，不能撑开侧栏。
- 移动端侧栏应优先显示草稿、当前对话和最近少量对话，历史入口折叠。

## 错误处理

- 草稿首发创建线程失败：保留草稿，保留用户输入，显示可重试错误。
- 线程创建成功但流式发送失败：保留真实线程，显示消息发送失败态，允许重试发送。
- `listThreads()` 失败：保留本地草稿入口和错误提示，不阻塞用户重新开始。
- 旧空线程折叠逻辑出错时，回退为展示原始线程列表，不影响发送。

## 验证标准

第一阶段完成后应满足：

1. 连续点击“开始新对话”10 次，左侧最多出现一个空白草稿。
2. 连续点击“开始新对话”10 次，在发送第一条消息前不会产生 10 次 `POST /api/v1/chat/threads`。
3. 草稿中发送第一条消息后，只创建一个真实线程，并正常进入流式回复。
4. 旧的重复空线程不会在左侧普通列表中铺满屏幕。
5. 有真实摘要或真实风险信息的历史对话不会被折叠成空白对话。
6. 50 条历史线程时，侧栏仍能稳定展示最近对话、折叠更早内容，并且文字不溢出。
7. 桌面和移动视口下，草稿项、搜索或查看更多入口不遮挡聊天输入区。

## 测试建议

- TypeScript 检查：`npm run check`。
- 构建检查：`npm run build`。
- 浏览器验证：
  - 桌面宽度下连点新对话，观察列表只保留一个草稿。
  - 移动宽度下打开侧栏，确认按钮和列表文字不溢出。
  - 使用网络面板确认草稿状态不会立即调用 `POST /threads`。
- 若新增纯函数文件，覆盖以下规则：
  - 重复疑似空线程只保留最新一个。
  - 有摘要的线程永远不被当作空线程折叠。
  - `L1` 及以上风险线程永远不被当作空线程折叠。

## 后续演进

第一阶段解决用户可见问题和新数据污染。第二阶段应补后端治理能力：

1. `ThreadListItem` 返回 `message_count`、`is_empty`、`archived_at`。
2. `GET /threads` 支持分页、搜索、归档筛选。
3. 提供归档接口，默认把更早低活跃线程移出左侧列表。
4. 提供安全的空线程清理脚本或管理接口，只处理 `message_count = 0` 的线程。
5. 前端历史抽屉使用后端分页，不一次性加载所有历史。

## Spec 自查

- 没有把空白对话删除作为默认行为，避免误伤心理咨询历史。
- 第一阶段不依赖后端新增字段，可以先落地。
- 第二阶段明确了后端字段和接口方向，用于治理已有堆积。
- “只能有一个新的空对话”由本地草稿单例保证，不依赖用户少点按钮。
- 大量对话堆积处理同时覆盖展示、搜索、归档和风险优先级。
