# 前端每日首聊建议设计

## 背景

右侧栏“可以试试”目前使用三条固定文案，并且点击后走 `handleQuickAction()` 的旧路径，直接调用 `POST /api/v1/chat/threads` 创建真实后端线程。这会绕过左侧已经实现的单一草稿模型，导致用户连续点击建议时仍然生成多条空对话。

用户期望这一区域不再是模板按钮，而是每天第一次进入对话时，根据已有上下文给出三条开启今天第一次对话的建议。建议只在当天出现一次；用户点击任意一条后，三条建议应同时消失。

## 目标

1. 右侧三条建议每天最多出现一次。
2. 点击任意建议后，整组建议立即消失，同一天刷新或重新进入也不再出现。
3. 点击建议不创建后端线程，不污染左侧历史列表。
4. 点击建议只激活现有本地草稿，并把选中的建议填入聊天输入框，真正发送时才沿用草稿首发创建线程流程。
5. 建议文案来自综合上下文，而不是固定三条模板。

## 上下文来源

当前前端可用的信息包括：

- 用户模式：青少年 / 成人。
- 记忆模式：关闭 / 摘要 / 长期记忆。
- 当前时间：白天 / 夜间。
- 今日心情是否已记录、最近一次心情分数。
- 情绪趋势：平均分、常见标签、日粒度记录。
- 每周小结：后端 `weekly-summary`，其中 `generated_by="llm"` 时代表后端已使用模型综合近 7 天情绪和会话摘要。

第一阶段不新增后端接口。前端优先使用后端 LLM 周小结与情绪趋势生成“今日首聊建议”；当后端只返回 fallback 或还没有足够数据时，使用本地安全兜底生成更柔和的开场建议。后续可以替换为专用后端接口，例如 `GET /api/v1/chat/daily-opening-suggestions`。

## 文案规则

- 不再使用旧三条固定文案：
  - “写下此刻最想被听见的一句话”
  - “用三句话卸下今晚的压力”
  - “试试 30 秒呼吸练习”
  - “把问题拆成一个很小的下一步”
  - “想想一个可以联系的可信任大人”
- 三条建议应像“用户可以直接写下的开场方向”，而不是功能说明或教学说明。
- 有情绪标签时，优先贴近标签，例如焦虑、疲惫、孤单、平静。
- 有 LLM 周小结时，允许用“顺着这周小结”作为上下文来源，但按钮本身不展示技术来源。
- 青少年模式可以更早提示现实支持，但不能变成警报式文案。
- 记忆关闭时，建议应给用户“只说此刻也可以”的边界感。

## 状态与幂等

新增本地存储标记：

```text
ningyu.dailyOpeningSuggestions.seenDay.<ownerId> = YYYY-MM-DD
```

规则：

- ownerId 优先使用当前用户 id，其次用户名，最后 `local`。
- 当当天第一次决定展示建议时，立即写入 seenDay；这样同一天刷新或重新进入不会重复出现。
- 当前这一次渲染中，建议会保持可见，直到用户点击其中一条。
- 点击任意建议后，当前页面立即隐藏整组建议，并再次写入当天 seenDay。
- 存储不可用时，仍允许当前页面展示建议，但无法跨刷新保证每日一次。

## 点击行为

点击建议时：

1. 不调用 `api.startThread()`。
2. 调用现有 `activateDraft()`，保持“只有一个本地空草稿”的规则。
3. 把建议标题作为聊天输入框草稿。
4. 清空快捷入口错误和旧反馈。
5. 隐藏整个右侧建议 section。

后端线程仍只在用户真正点击发送时由 `ensureThreadForSend()` 创建。

## 验证标准

1. 清空当天 `ningyu.dailyOpeningSuggestions.seenDay.*` 后进入主界面，右侧出现三条今日首聊建议。
2. 三条建议不是旧固定模板。
3. 点击任意一条后，三条建议全部消失。
4. 点击建议时网络面板不出现 `POST /api/v1/chat/threads`。
5. 点击建议后左侧最多只有一个“当前草稿”。
6. 输入框出现被点击的建议文案。
7. 刷新同一天页面，建议不再出现。
