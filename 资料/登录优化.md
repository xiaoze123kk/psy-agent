## 2. 阶段一：P0 安全加固

> 提交：`fix: 加固登录模块安全性 — 密码校验、限流审计、Token 轮转原子性`

### 2.1 密码强度校验

对密码做四重规则校验，分别返回明确错误信息，而非模糊的"密码不合法"。

```python
# backend/app/core/security.py
def validate_password_strength(password: str) -> None:
    if len(password) < MIN_PASSWORD_LENGTH:  # 8
        raise ValueError("密码至少需要 8 个字符。")
    if not any(c.isupper() for c in password):
        raise ValueError("密码需要包含至少一个大写字母。")
    if not any(c.islower() for c in password):
        raise ValueError("密码需要包含至少一个小写字母。")
    if not any(c.isdigit() for c in password):
        raise ValueError("密码需要包含至少一个数字。")
```

### 2.2 登录频率限制 + 审计日志

**服务**：`backend/app/services/login_attempt_service.py`（新增）

- 基于内存的计数器（单进程安全）
- 同一 IP + 用户名 组合在 5 分钟内失败超过 5 次 → 临时封禁 15 分钟
- 每次失败记录 `reason`（`invalid_credentials`、`blocked` 等）

**端点集成**（`auth.py`）：

```
POST /api/v1/auth/login
  → 检查 is_blocked(ip, username) → 429
  → 验证密码失败 → record_failure(ip, username, reason) → 401
  → 验证通过 → clear(ip, username) → 正常签发 token
```

### 2.3 Refresh Token 轮转原子性

**先签发后撤销**，单 `commit` 保证原子性：

```python
# _issue_token_pair → DB 写入新 refresh_token
# _revoke_refresh_token → 标记旧 token 为 rotated
db.commit()  # 一次 commit 完成新旧交替
```

### 2.4 注册流程事务安全

移除中间的 `db.flush()` 调用，所有对象（User、UserProfile、UserSettings、RefreshToken）一次 `db.add_all()` + 单次 `db.commit()`。

## 3. 阶段二 P1.3a：忘记密码全流程

> 提交：`feat: 忘记密码完整流程（密保验证 + 密码重置）`

### 3.1 数据库扩展

**UserProfile 表**新增两列：

| 列 | 类型 | 用途 |
|----|------|------|
| `security_question` | `VARCHAR(200)` | 密保问题，注册时设置 |
| `security_answer_hash` | `TEXT` | 密保答案的 PBKDF2 哈希 |

**PasswordResetToken 表**（新建）：

| 列 | 类型 | 用途 |
|----|------|------|
| `id` | UUID PK | token 唯一标识 |
| `user_id` | UUID FK | 关联用户 |
| `token_hash` | `VARCHAR(64)` | 令牌的 SHA-256 哈希 |
| `status` | `VARCHAR(16)` | active / used / revoked |
| `expires_at` | `TIMESTAMPTZ` | 5 分钟过期 |
| `used_at` | `TIMESTAMPTZ` | 使用时间 |

### 3.2 前端三步流程

```
第一步：输入用户名
    → GET /api/v1/auth/password-reset-question?username=xxx
    → 展示该用户关联的密保问题

第二步：回答密保问题
    → POST /api/v1/auth/password-reset-verify { username, answer }
    → 正确 → 返回 reset_token（有效期 5 分钟）

第三步：重置密码
    → POST /api/v1/auth/password-reset { reset_token, new_password }
    → 成功后吊销该用户所有活跃 refresh_token（踢出所有会话）
    → 切回登录页
```

### 3.3 安全性

- 密保答案使用 PBKDF2 哈希存储，不存明文
- reset_token 使用与 access/refresh token 相同的 HMAC 签名算法，有效期为 5 分钟
- 密码重置后自动吊销所有活跃 refresh_token
- 注册时必须填写密保问题和答案（预防用户遗忘后无法找回）

## 4. 阶段二 P1.3b：7天自动登录（双 Token 机制）

### 4.1 需求概要

| 场景 | 未勾选"7天自动登录" | 勾选"7天自动登录" |
|------|-------------------|-----------------|
| 最后一次登录后多久需重新输入密码 | 1 小时 | 7 天 |
| 已登录状态下刷新页面 | 保持登录 | 保持登录 |
| 关闭页面后再打开（短时间内） | 1 小时内免登录 | 7 天内免登录 |
| 超时后 | 需重新登录 | 需重新登录 |

### 4.2 为什么需要双 Token

| | access_token | refresh_token |
|---|---|---|
| 定位 | 证明"我已登录"，访问受保护资源 | 证明"我之前登录过"，换取新 access_token |
| 有效期 | 固定 1 小时 | 未勾选 1 小时 / 勾选 7 天 |
| 前端存储 | JS 内存（`tokenStore._accessToken`） | 浏览器 Cookie（`rt`, HttpOnly） |
| 后端存储 | 无（纯 JWT 验证） | DB `refresh_tokens` 表 |
| 可撤销 | 否（JWT 无状态） | 是（DB 标记 `status="revoked"`） |
| 使用频率 | 每个 API 请求 | 仅在 access_token 过期时 |
| 传输方式 | `Authorization: Bearer <token>` 请求头 | 浏览器自动带 Cookie |
| JWT payload | `{sub, type:"access", exp}` | `{sub, type:"refresh", exp, jti}` |

### 4.3 Token 生命周期

#### 签发

```python
_issue_token_pair(db, user, auto_login)
  → access_token TTL = 3600s（固定）
  → refresh_token TTL = 3600s（未勾选）或 604800s（勾选）
  → 设置 HttpOnly Cookie: rt=<refresh_token>; Path=/api/v1/auth; SameSite=Lax
  → 写入 DB refresh_tokens 表
```

#### 过期时间的三层控制

三层控制完全同步：

| 控制层 | 机制 |
|--------|------|
| **Cookie Max-Age** | 浏览器自动清除 |
| **JWT exp** | `decode_token` 验证 |
| **DB expires_at** | `_validate_refresh_token` 检查 |

#### 请求流程

```
正常请求：
  前端 → Authorization: Bearer <access_token>
  后端 → decode_token(token, "access") → 200

access_token 过期：
  后端 → 401
  前端 → refreshAuthToken()
       → POST /api/v1/auth/refresh（浏览器自动带 rt Cookie）
       → 后端验证 → 签发新 pair → Set-Cookie 新 rt
       → 前端收到新 access_token → 重试原始请求

refresh_token 过期：
  前端 → POST /api/v1/auth/refresh → 401
        → session.status = "anonymous" → 登录页
```

#### Token 轮转（Rotation）

每次 `/refresh` 旧 token 立即作废：

```
旧 refresh_token: status="active" → status="rotated", revoked_at=now
新 refresh_token: status="active", 新 jti, 新 expires_at
```

攻击者重放旧 token → token 已标记 rotated → 401

### 4.4 撤销场景

| 场景 | 操作 |
|------|------|
| 用户主动注销 | `POST /logout` → DB 标记 revoked，清除 Cookie |
| Token 轮换 | `POST /refresh` → 旧 token 标记 rotated |
| 密码重置 | 遍历所有 active token → 全部 revoked |
| Token 过期 | 自动标记 revoked |

### 4.5 页面刷新时的会话恢复

```
F5 刷新页面
  │
  ├─ JS 内存清空 → tokenStore._accessToken = null
  ├─ Cookie 仍在（rt），localStorage 仍在
  │
  ▼
restoreSession()
  ├─ POST /api/v1/auth/refresh（靠 Cookie，不带 Authorization）
  ├─ 成功 → 进入主应用
  └─ 失败 → 显示登录页（用户名预填）
```

### 6.3 P1.4 高并发验证码 TODO 标记

**现状**：`captcha_service.py` 使用内存 `dict` 存储验证码挑战：

```python
class CaptchaStore:
    def __init__(self) -> None:
        self._challenges: dict[str, CaptchaChallenge] = {}  # 内存存储
```

**问题**：

| 部署模式 | 安全性 | 原因 |
|---------|--------|------|
| 单进程（当前 dev） | ✅ 安全 | GIL 保护 `dict.pop()`，无 `await` 插入点 |
| 同 `captcha_id` 并发验证 | ✅ 安全 | `pop()` 只被第一个请求消费 |
| 多进程/多 worker | ❌ 不工作 | 每个 worker 独立内存，worker A 创建的验证码 worker B 不认识 |

**方案**：生产环境部署多 worker 时，将 `CaptchaStore` 从内存 dict 迁移至 Redis：

```python
# 迁移后伪代码
class CaptchaStore:
    def __init__(self, redis_client):
        self._redis = redis_client
        self._prefix = "captcha:"

    def create(self) -> dict:
        challenge_id = str(uuid4())
        answer = "".join(secrets.choice(CAPTCHA_CHARS) for _ in range(5))
        key = f"{self._prefix}{challenge_id}"
        self._redis.setex(key, CAPTCHA_TTL_SECONDS, self._hash_answer(challenge_id, answer))
        return {"captcha_id": challenge_id, "image_data_url": self._render_svg_data_url(answer), "expires_in": CAPTCHA_TTL_SECONDS}

    def verify(self, challenge_id: str, code: str) -> bool:
        key = f"{self._prefix}{challenge_id}"
        stored_hash = self._redis.get(key)
        if stored_hash is None:
            return False
        self._redis.delete(key)
        return hmac.compare_digest(stored_hash, self._hash_answer(challenge_id, code))
```

**任务**：在 `captcha_service.py` 文件顶部添加 TODO 注释，标注多 worker 部署时需迁移至 Redis。

**动机**：当前无法一键吊销用户所有 token（除非改 `secret_key`，但会影响所有用户）。需要引入 token 版本号机制。

if "token_version" not in user_columns:

**签发** — [security.py](file:///d:/test/--agent/----agent/backend/app/core/security.py) `create_access_token` / `create_refresh_token` 接受 `token_version` 参数，写入 payload `ver` 字段：

```python
def create_token(..., *, token_version: int | None = None) -> str:
    payload = {"sub": ..., "type": ..., "exp": ...}

        payload["ver"] = token_version
```

**验证** — [deps.py](file:///d:/test/--agent/----agent/backend/app/api/deps.py) `get_current_user` 校验：

```python
if payload.get("ver") != user.token_version:
    raise HTTPException(status_code=401, detail="Token has been revoked.")

**吊销** — [auth.py](file:///d:/test/--agent/----agent/backend/app/api/v1/endpoints/auth.py) 新增 `_revoke_all_user_tokens()` 辅助函数，`change_password` 中 `token_version += 1`：

```python
def _revoke_all_user_tokens(db: Session, user: User) -> None:
    active_tokens = db.scalars(
        select(RefreshToken).where(RefreshToken.user_id == user.id, RefreshToken.status == "active")
    ).all()
    for rt in active_tokens:
        _revoke_refresh_token(rt, status_value="revoked")
```

**复用**：`_revoke_all_user_tokens` 同时用于 `password_reset` 和 `change_password`，消除重复代码。

### 7.2 P2.2 并发会话上限

> 提交：`feat: 并发会话上限`

**动机**：当前没有限制每个用户可同时持有的活跃 refresh_token 数量。用户可以无限登录，产生大量有效 token，且不便于管理。

**实现**：在 `_issue_refresh_token()` 中签发新 token 前检查活跃数，超限则自动吊销创建时间最早的 token。

```python
MAX_ACTIVE_SESSIONS = 5

active_count = db.scalar(
    select(func.count(RefreshToken.id)).where(
        RefreshToken.user_id == user.id,
        RefreshToken.status == "active",
    )
)
if active_count and active_count >= MAX_ACTIVE_SESSIONS:
    oldest = db.scalars(
        select(RefreshToken)
        .where(RefreshToken.user_id == user.id, RefreshToken.status == "active")
        .order_by(RefreshToken.created_at.asc())
        .limit(1)
    ).first()
    if oldest:
        _revoke_refresh_token(oldest, status_value="revoked")
```

**行为**：用户最多同时持有 5 个活跃会话。第 6 次登录时，最早会话自动下线。

**涉及文件**：`backend/app/api/v1/endpoints/auth.py`（常量 + `_issue_refresh_token` 逻辑）

### 7.3 P2.3 密码修改功能

> 提交：`feat: 密码修改功能`

**动机**：当前用户只能通过"忘记密码"流程重置密码，没有常规密码修改入口。

**实现**：

**新增端点**：

```
POST /api/v1/auth/change-password
  Request:  { old_password, new_password }
  Response: ChangePasswordResponse（含新 token pair + 完整用户信息）
```

**后端逻辑**：

```
1. JWT 身份认证（get_current_user）
2. 验证 old_password 与数据库匹配 → 400
3. validate_password_strength(new_password) → 422
4. 更新 user.password_hash
5. `user.token_version += 1`（全局吊销所有已签发 token）
6. 吊销所有活跃 refresh_token（其他设备强制下线）
7. 签发新 token pair → Set-Cookie 新 rt
8. 返回 ChangePasswordResponse（用户无需重新登录）
```

**涉及文件**：

| 文件 | 改动 |
|------|------|
| `backend/app/schemas/auth.py` | 新增 `ChangePasswordRequest`、`ChangePasswordResponse` |
| `backend/app/api/v1/endpoints/auth.py` | 新增 `POST /change-password` 端点（含新旧密码校验、session 吊销、新 token 签发） |
| `frontend/src/types/api.ts` | 新增接口（留好前端接口） |
| `frontend/src/api/endpoints.ts` | 新增 `changePassword()` 方法（留好前端接口） |
