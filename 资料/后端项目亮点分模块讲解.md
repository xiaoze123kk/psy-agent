# 后端项目亮点分模块讲解

> 关联笔记：[[项目模块讲解与LangGraph节点图]]
> 适用场景：给自己梳理架构、给别人讲项目亮点、准备答辩/面试/重构后交接。
> 重点：这篇不按文件树讲，而是按“后端能力模块”讲。

## 总览

这个后端的亮点不是“能调用大模型聊天”，而是围绕心理咨询陪伴场景，把一轮对话拆成了多个可控模块：

- 风控系统决定当前对话是否安全、是否需要危机路径、是否要阻断普通 RAG。
- 控制面决定当前轮应该走危机、红旗、边界、澄清，还是普通支持。
- RAG 系统给模型提供咨询语料和知识参考，但高风险时会自动收缩。
- 记忆系统负责会话连续性和长期用户理解，但会按风险、可见性和用户设置控制。
- 对话走向策略负责“这轮应该怎么接话”，避免每轮都模板化咨询。
- 内容校验系统负责安全和体验出口检查，必要时触发重写。
- Graph trace 和质量 trace 让每一轮为什么这样回复可以被追踪。
- Chat service 把 LangGraph、数据库、幂等、流式输出、失败兜底串成一个稳定事务。

可以用一句话概括：

> 这是一个以心理安全、体验质量、上下文连续性和可观测性为中心的 LangGraph 后端，而不是一个薄封装的大模型代理。

## 后端亮点速览表

| --- | --- | --- | --- |

| RAG 系统 | `rag_nodes.py`、`counseling_vector_service.py`、`counseling_reranker.py`、`knowledge_service.py` | 检索知识和咨询语料 | 咨询语料按过程片段建模，支持 rerank、配额、trace 和安全 gating |
| 记忆系统 | `memory_service.py`、`memory_nodes.py`、`memory_job_service.py` | 长期连续性和用户画像 | 读写分离、风险可见性控制、相似度去重、后台写入 |

| 内容校验 | `validator_nodes.py`、`validator_experience.py` | 输出前安全和体验检查 | 安全硬规则 + 体验软规则，失败可重写 |
| 可观测性 | `graph_trace_service.py`、`conversation_quality_service.py` | 解释每轮为什么这样回复 | 保存 graph trace、RAG trace、validator reasons、quality trace |
| 流式与幂等 | `chat_streaming.py`、`chat_turn_lifecycle.py` | 用户体验和重复请求安全 | SSE 分阶段事件、turn claim/replay、失败状态可恢复 |
| 隐私与数据治理 | `privacy_service.py`、`UserMemory` 相关表 | 敏感信息和用户控制 | 记忆可见性、隐私操作日志、风险场景下限制上下文扩散 |

## 1. 聊天事务编排

### 它解决什么问题

用户点击发送消息，看起来只是一次聊天。但后端真正要保证的是：

- 这条消息属于当前用户和当前会话。
- 重复点击或网络重试不会生成两次不同回复。
- 大模型或 LangGraph 超时要有兜底。
- 用户消息、assistant 消息、trace、风险事件、记忆任务要一起处理。
- 流式接口要能持续给前端反馈，而不是卡住。

这些职责集中在 `app/services/chat_service.py`，它是后端“一轮对话事务”的主入口。

### 核心链路

非流式：

```text
API chat.py
  -> process_message_turn
  -> claim_turn
  -> create user message
  -> prepare turn context
  -> graph_runtime.invoke_turn
  -> persist assistant result
  -> complete_turn
  -> return response
```

流式：

```text
API chat.py
  -> process_message_turn_stream
  -> accepted event
  -> graph_update / token / heartbeat events
  -> persist final result
  -> final event
```

### 为什么是亮点

它把“大模型聊天”变成了一个可恢复、可追踪、可持久化的业务事务。很多 agent demo 只关心生成文本，但这个项目关心的是生产级聊天会遇到的问题：

- 重复请求如何处理。
- 图失败时怎么返回。
- trace 怎么保存。
- 记忆写入是否要异步。
- 流式输出中途失败怎么办。

这让后端不是薄薄一层 wrapper，而是有稳定运行边界的对话服务。

## 2. LangGraph 编排

### 它解决什么问题

心理咨询 agent 不能把所有逻辑塞进一个 prompt。不同阶段有不同职责：

- 输入要清洗。
- 风险要优先判断。
- 控制面要决定能不能走普通支持。
- RAG 要在合适场景才启用。
- 回复生成要按场景分流。
- 输出要被校验。
- 最后要总结和决定是否写记忆。

这些由 `app/graphs/main_graph.py` 编成一张主图。

### 主图结构

```text
normalize_input
  -> load_user_profile
  -> risk_classifier
  -> control_plane
  -> intent_classifier / crisis_response / boundary_response / clarification_response
  -> example_retriever
  -> response node
  -> response_validator
  -> summarize_turn
  -> memory_candidate_extract
  -> write_memory / skip_memory
```

节点图见：

![[assets/langgraph-main-graph.svg]]

### 为什么是亮点

主图把“心理安全优先”和“普通聊天体验”分层了。不是所有请求都进入同一个模型 prompt，而是先经过风险和控制面：

- 高风险直接去 `crisis_response`。
- 临床红旗去 `clinical_red_flag_response`。
- 边界/系统保护去 `boundary_response`。
- 信息不足去 `clarification_response`。
- 普通支持才进入意图识别和 RAG。

这种分流降低了模型在高风险场景乱发挥的概率，也让每个节点能专注自己的职责。

## 3. 风控系统

### 它解决什么问题

心理咨询类产品必须优先处理安全风险。风控系统要回答：

- 用户有没有自伤、伤人、受害、临床红旗等风险？
- 风险是模糊、近期、主动还是无风险？
- 当前轮能不能继续普通聊天？
- 是否需要安全检查、危机资源、低压微行动？
- 是否要阻断 RAG、普通记忆引用或工具调用？

相关代码：

- `app/graphs/nodes/risk_nodes.py`
- `app/graphs/nodes/control_nodes.py`
- `app/services/risk_policy.py`
- `app/services/safety_context_service.py`

### risk_classifier

`risk_classifier` 会输出：

- `risk_level`：`L0`、`L1`、`L2`、`L3`
- `risk_domain`
- `immediacy`
- `risk_confidence`
- `protective_signals`
- `risk_reason_codes`
- `risk_response_policy`

它不是只看一个关键词，而是把关键词信号、语义风险、原因码和可选 LLM refine 结合起来。

### control_plane

`control_plane` 是风险之后的控制面。它会输出：

- `route_priority`
- `control_category`
- `clarification_needed`
- `response_contract`
- `memory_policy`
- `rag_policy`
- `tool_gate_mode`

路由优先级很关键：

| 优先级 | 含义 | 后续节点 |
| --- | --- | --- |
| `P0_immediate_safety` | 立即安全优先 | `crisis_response` |
| `P1_red_flag` | 临床红旗 | `clinical_red_flag_response` |
| `P2_support` | 普通支持 | `intent_classifier` |
| `P3_bridge_boundary` | 需要桥接边界 | 多数仍进入受限支持 |
| `P4_system_protection` | 系统保护/边界 | `boundary_response` |

### 为什么是亮点

风控系统不是只在输出后过滤一句“不能回答”。它是全链路控制：

- 影响路由。
- 影响 prompt contract。
- 影响 RAG 是否启用。
- 影响记忆是否可见。
- 影响工具是否可用。
- 影响最终回复按钮。

这让安全策略前置，而不是事后补救。

### 可讲的技术点

如果要向别人介绍，可以这样说：

> 项目把心理风险控制做成了图里的早期节点。风险等级不只是一个标签，而会改变后续路由、RAG、记忆引用和回复契约。这样高风险场景不会误入普通咨询或闲聊流程。

## 4. RAG 系统

### 它解决什么问题

RAG 在这个项目里有两类用途：

1. 知识库 RAG：用于心理健康知识内容。
2. 咨询语料 RAG：用于给回复生成提供咨询过程和语言风格参考。

最有特色的是咨询语料 RAG。它不是为了“查知识答案”，而是为了让模型参考更像真实咨询对话的回应方式。

相关代码：

- `app/graphs/nodes/rag_nodes.py`
- `app/services/counseling_vector_service.py`
- `app/services/counseling_reranker.py`
- `app/services/counseling_examples.py`
- `app/services/counseling_chunking.py`
- `app/services/knowledge_service.py`
- `app/services/embedding_service.py`
- `app/services/milvus_service.py`

### 咨询语料 chunk 设计

咨询语料不是简单按固定字数切片，而是按对话意义分层：

| chunk 类型 | 说明 | 用途 |
| --- | --- | --- |
| `turn_pair` | 单个 user-assistant 轮次 | 学局部回应方式、措辞和节奏 |
| `process_segment` | 3-5 轮咨询片段 | 学情绪变化和咨询师推进方式 |
| `session_sketch` | 整段咨询地图 | 学主题、情绪线、干预路径，不保存逐字原文 |

这个设计非常适合心理咨询场景，因为咨询不是知识问答，而是过程性互动。

### 检索链路

```text
example_retriever
  -> counseling_rag_allowed
  -> embedding query
  -> Milvus recall
  -> local/model rerank
  -> quota selection
  -> retrieved_counseling_examples
  -> prompt builder
```

### 安全 gating

RAG 有明确的安全边界：

- `risk_level` 是 `L2` / `L3` 时阻断咨询语料 RAG。
- Milvus 不可用时不会阻断聊天，而是返回 fallback。
- embedding 不可用时会跳过 RAG。
- reranker 失败时会退回确定性选择。
- retrieved examples 会被 validator 防复制，避免 RAG 原文泄露。

### rerank 亮点

`counseling_reranker.py` 支持：

- 本地 reranker worker。
- 超时控制。
- reranker disabled/unavailable fallback。
- scored_count、duration、reason trace。
- 按 query 类型做 chunk 配额选择。

这意味着 RAG 不是“搜到什么塞什么”，而是有召回、重排、配额、失败降级和 trace 的完整链路。

### 为什么是亮点

很多项目的 RAG 是知识检索。这个项目的咨询 RAG 更像“过程参考检索”：它检索的不是标准答案，而是咨询对话片段，用来改善回复的语气、节奏和干预路径。

可以这样介绍：

> 这个 RAG 系统不只检索知识，还检索咨询过程样本。语料按 turn_pair、process_segment、session_sketch 三层建模，并且高风险时自动阻断，reranker 失败时降级，trace 中能看到每次召回和重排状态。

## 5. 记忆系统

### 它解决什么问题

心理陪伴类 agent 需要连续性，但记忆又非常敏感。记忆系统要解决两组相反的需求：

- 要记得用户的重要偏好、目标、反复出现的困扰、有效支持方式。
- 不能无节制保存敏感内容，不能在高风险场景乱引用记忆，不能让过期记忆误导当前回复。

相关代码：

- `app/services/memory_service.py`
- `app/services/memory_scoring.py`
- `app/services/memory_job_service.py`
- `app/graphs/nodes/memory_nodes.py`
- `app/services/user_context_pack_service.py`
- `app/services/user_context_service.py`

### 记忆类型

记忆不是一团文本，而是有类型的。常见类型包括：

- `session_summary`
- `goal`
- `state`
- `preference`
- `correction`
- `support_strategy`
- `recurring_trigger`
- `relationship`
- `profile`
- `safety_summary`

这些类型会影响检索优先级、可见性和是否能在高风险场景使用。

### 读记忆链路

聊天前，`chat_service` 会调用：

```text
build_memory_index
retrieve_memories_for_turn_async
build_user_context_pack
build_safety_context_pack
```

记忆检索会综合：

- 用户记忆模式：`off`、`summary_only`、`long_term`
- 当前风险等级
- 记忆类型
- 记忆可见性
- 文本相似度
- 向量分数
- 重要性
- 新鲜度
- access count
- 当前 goal_state 和 control_category

### 写记忆链路

LangGraph 尾部会执行：

```text
summarize_turn
  -> memory_candidate_extract
  -> write_memory / skip_memory
```

`memory_candidate_extract` 会根据当前状态决定是否生成候选记忆。常见策略：

- `failed_no_reply`：不写。
- `memory_mode == off`：不写。
- `skip_sensitive`：不写。
- `crisis_audit_only`：只生成安全摘要类记忆。
- `summary_only`：只保留会话摘要。
- `long_term`：提取结构化长期记忆候选。

业务层还会把真正写入包装成 `PendingMemoryJob`，由后台 worker 处理，避免阻塞聊天主链路。

### 评分与去重

`memory_scoring.py` 负责纯文本评分辅助：

- 清洗文本。
- tokenize。
- term similarity。
- content similarity。
- 判断两个记忆是否值得比较。

`memory_service.py` 在此基础上叠加业务评分：

- 类型 boost。
- 目标相关 boost。
- 冲突/纠偏相关 boost。
- 重要性。
- 新鲜度。
- 访问次数。
- vector score。

### 高风险场景下的记忆控制

这是记忆系统最值得讲的地方之一。

当风险等级是 `L2` / `L3` 时，系统不会像普通场景那样自由引用长期记忆，而是：

- 限制可见记忆类型。
- 允许 `internal_safety` 的 `safety_summary`。
- 控制 `user_visible` 记忆是否能进入当前轮。
- 不在危机场景暴露不必要的个人历史。

### 为什么是亮点

这个记忆系统不是简单“把用户说过的话存起来”。它把记忆当作受风险、隐私、类型、时效和对话目标共同约束的上下文资产。

可以这样介绍：

> 记忆系统同时支持长期连续性和安全边界。它按记忆类型、可见性和风险等级控制读写，用规则分数加向量分数排序，并通过后台任务写入，避免聊天主链路被记忆持久化拖慢。

## 6. 对话走向策略

### 它解决什么问题

很多心理陪伴 agent 的问题不是“不安全”，而是“不像人在对话”：

- 每轮都“听起来你……”。
- 每轮都分析成创伤、压抑、防御。
- 用户说“别分析我”，下一轮还继续分析。
- 用户提了一个具体作品/人物/隐喻，回复却泛化成普通情绪。
- 问题太多，像问卷。

这个项目用 `conversation_move_policy` 来处理这些体验层问题。

相关代码：

- `app/services/conversation_move_policy.py`
- `app/services/conversation_policy_anchors.py`
- `app/services/conversation_policy_adaptation.py`
- `app/services/conversation_policy_structure.py`

### 主要输出

它会生成 `conversation_move_policy`，其中包括：

- `conversation_move`
- `intent_lanes`
- `primary_lane`
- `ningyu_voice_contract`
- `adaptation_state`
- `topic_anchor`
- `anchor_value`
- `anchor_evidence`
- `structure_mode`
- `structure_style`
- `psychologizing_risk`
- `button_style`

这些字段会影响 prompt 和 validator。

### 锚点识别

`conversation_policy_anchors.py` 会识别：

- 文学作品，如《德米安》《在轮下》。
- 哲学人物或概念，如荣格、尼采、存在主义。
- 影视/媒体角色。
- 人物 anchor。
- 日常细节，如花、包子、咖啡。
- 隐喻，如“被推着走”“奔跑”“碾死”。

识别到锚点后，系统会更倾向于直接回应用户提到的具体东西，而不是泛泛地说“你很难受”。

### 短期适配

`conversation_policy_adaptation.py` 会读取最近 assistant metadata 和用户纠偏信号，例如：

- “别分析我”
- “不要一直问”
- “别问安全”
- “不是这个意思”
- explicit feedback：too_analytic、too_many_questions、too_generic、missed

然后生成短期状态：

- `avoid_analysis_turns`
- `avoid_questions_turns`
- `avoid_safety_check_turns`
- `prefer_direct_anchor_response_turns`

这让系统能在被用户纠正后真正改变，而不是只道歉一句。

### 回复结构变化

`conversation_policy_structure.py` 会检测最近回复结构，避免连续复用：

- 两段式整理 + 追问。
- 单段。
- 短答。
- 停顿式邀请。

这样能减少“每轮都一个格式”的机械感。

### 为什么是亮点

这个模块把“好的对话手感”编码成了可解释策略。它不是安全必须项，但非常影响用户是否愿意继续聊。

可以这样介绍：

> 项目单独做了一层 conversation move policy，用来控制每一轮的对话动作、声线、问题预算、分析深度和结构变化。它让 agent 不只安全，还更会接话。

## 7. 内容校验系统

### 它解决什么问题

大模型输出不能直接给用户。尤其在心理咨询场景，要检查：

- 是否泄露诊断。
- 是否给用药或危险方法。
- 是否承诺绝对保密。
- 是否冒充真人/医生/咨询师。
- 是否强化依赖。
- 是否复制 RAG 原文。
- 是否过度心理化。
- 是否问题太多。
- 是否忽略用户当前锚点。
- 是否胡编文化作品细节。

相关代码：

- `app/graphs/nodes/validator_nodes.py`
- `app/graphs/nodes/validator_experience.py`

### 两类校验

#### 1. 安全校验

位于 `validator_nodes.py`，检查硬性安全问题：

- diagnosis leak
- medication leak
- dangerous method
- delusion confirmation
- dependency reinforcement
- treatment promise
- absolute confidentiality
- role impersonation
- unverified resource
- unsafe button
- RAG copy leak

这些通常是 blocking。

#### 2. 体验校验

位于 `validator_experience.py`，检查体验质量：

- banned phrase。
- moralizing reassurance。
- professional referral first turn。
- length budget exceeded。
- too many questions。
- repeated safety question。
- reused formulaic opening。
- ignored topic anchor。
- over psychologizing。
- generic buttons。
- conversation restart。
- fabricated cultural claim。
- overconfident cultural claim。
- reused reply structure。
- missed primary lane。
- violated voice contract。
- failed short-term adaptation。

体验校验会区分 warning 和 block。

### 重写机制

`response_validator` 如果发现 blocking reasons，会尝试调用模型重写：

```text
validator failed
  -> build repair prompt
  -> deepseek_client.chat
  -> parse reply/actions
  -> re-run validator
  -> pass / failed_no_reply
```

如果重写仍失败，就返回 `failed_no_reply`，避免把有问题的回复发给用户。

### 为什么是亮点

内容校验不是单纯关键词过滤，而是结合了：

- 安全策略。
- conversation move policy。
- 用户近期纠偏。
- 文化锚点证据。
- 回复结构复用。
- RAG 防复制。

可以这样介绍：

> validator 同时做安全闸门和体验闸门。它不只是拦危险词，还会检查过度心理化、追问过多、忽略用户锚点、胡编作品细节等心理陪伴产品中特有的问题。

## 8. Graph trace 与质量追踪

### 它解决什么问题

当用户说“为什么这一轮回复这么奇怪”，或者开发者调试 RAG/风控/校验时，不能只看最终文本。需要知道：

- 哪个节点运行了。
- 每个节点输出了什么。
- RAG 有没有用。
- 为什么 RAG 被跳过。
- validator 有没有 warning/block。
- 记忆有没有写入。
- 最终 delivery_status 是什么。

相关代码：

- `app/services/graph_trace_service.py`
- `app/services/conversation_quality_service.py`
- `app/services/graph_runtime.py`
- `conversation_turn_traces` 表

### trace 类型

项目里有几类重要追踪：

| trace | 说明 |
| --- | --- |
| graph trace | LangGraph 每个节点的更新记录。 |
| trace summary | 给前端/API 使用的压缩摘要。 |
| RAG trace | 检索是否命中、召回数量、rerank 状态、耗时。 |
| quality trace | 对话质量评估，包括用户信号、validator 结果、是否重写。 |
| memory decisions | 记忆写入或跳过原因。 |

### 为什么是亮点

agent 产品最怕“黑箱”。这个项目把每轮对话中关键中间状态都留了下来，便于定位：

- 是风险分级错了？
- 是 control plane 走错路了？
- 是 RAG 没召回？
- 是 prompt 没接住 conversation move policy？
- 是 validator 修复失败？

这对迭代心理咨询 agent 很重要，因为体验问题往往不是单点 bug，而是链路里某个策略影响了后续节点。

## 9. 流式输出与幂等机制

### 它解决什么问题

聊天产品需要低等待感，也要处理网络重试。

相关代码：

- `app/services/chat_streaming.py`
- `app/services/chat_turn_lifecycle.py`
- `app/services/chat_service.py`
- `conversation_turns` 表

### 流式事件

SSE 流式接口会输出：

- `accepted`：后端已经接受本轮 turn。
- `graph_update`：风险、记忆检索、validator、保存等阶段更新。
- `token`：assistant 文本 token/chunk。
- `heartbeat`：图执行较久时告诉前端仍在运行。
- `final`：最终完整结果。
- `error`：流式失败。

### 幂等机制

`chat_turn_lifecycle.py` 负责：

- claim turn。
- 判断 replay。
- request hash。
- running turn 等待。
- complete turn。
- failed turn。

这样前端重试时，不一定会重新跑整张图，而是可以复用已有 turn 结果。

### 为什么是亮点

这让聊天体验更像真实产品：

- 用户不会盯着空白等。
- 后端慢的时候有 heartbeat。
- 网络重复提交不会轻易产生重复 assistant 回复。
- 失败状态能被记录和恢复。

## 10. 隐私与数据治理

### 它解决什么问题

心理咨询数据高度敏感。项目需要让用户数据可控、可追踪、可清理。

相关代码：

- `app/services/privacy_service.py`
- `app/schemas/privacy.py`
- `app/api/v1/endpoints/privacy.py`
- `privacy_action_logs` 表
- `user_memories` 的 `visibility`、`review_state`、`status`

### 主要设计点

- 记忆有 `visibility`，区分用户可见和内部安全摘要。
- 记忆有 `review_state`，可标记需要 review。

### 为什么是亮点

这个项目没有把“记忆”当成单纯便利功能，而是把它放在隐私和风险约束里。尤其在心理咨询场景，这是必要的产品底线。

可以这样介绍：

> 项目把记忆和隐私绑定起来：记忆有可见性、审核状态和操作日志，高风险时限制上下文扩散，向量库只是可重建索引，真正数据治理以 PostgreSQL 为准。

## 15. 可以对外讲的亮点版本

如果要在面试、答辩或项目介绍里讲，可以压缩成这几段。

### 风控亮点

> 我们把心理风险控制做成了 LangGraph 早期节点。风险等级会影响路由、RAG、记忆、工具和 prompt contract。高风险不会进入普通聊天或咨询语料 RAG，而是走危机安全路径。

### RAG 亮点

> RAG 不只是知识检索，还包括咨询语料过程检索。语料按 turn_pair、process_segment、session_sketch 分层，支持 Milvus 召回、本地 reranker、配额选择、失败降级和 trace。

### 记忆亮点

> 记忆系统按类型、可见性、风险等级和用户记忆模式控制读写。检索时结合规则分、向量分、重要性、新鲜度和目标相关性；写入通过候选提取和后台 job，避免主链路阻塞。

### 校验亮点

> 回复前有安全和体验双校验。除了危险内容，还检查问题过多、过度心理化、忽略用户锚点、文化作品胡编、重复模板结构等心理陪伴产品特有问题，失败时可触发重写。

### 可观测性亮点

> 每轮对话保存 graph trace、RAG trace、validator reasons、quality trace 和 memory decisions。这样可以解释一次回复为什么这样生成，也方便定位策略链路中的问题。

### 对话体验亮点

> conversation move policy 独立控制当前轮的对话动作、声线、问题预算、分析深度和结构变化。它能根据用户“别分析我”“别一直问”等纠偏信号做短期适配，减少模板感。

## 结尾总结

这个后端最值得保留和继续打磨的，不是某一个单点模型能力，而是这些模块之间形成的系统能力：

```text
风险控制
  -> 控制面路由
  -> 安全的 RAG / 记忆上下文
  -> 场景化回复生成
  -> 内容与体验校验
  -> 摘要和记忆沉淀
  -> trace 可解释闭环
```

这条链路让它更像一个面向心理咨询陪伴的工程化 agent，而不是一个普通聊天机器人。
