# 风控系统

> 返回：[[00 后端项目亮点分模块索引]]
> 相关代码：`backend/app/graphs/nodes/risk_nodes.py`、`backend/app/graphs/nodes/control_nodes.py`、`backend/app/services/risk_policy.py`、`backend/app/services/safety_context_service.py`

## 它解决什么问题

心理咨询类产品必须优先处理安全风险。风控系统要回答：

- 用户有没有自伤、伤人、受害、临床红旗等风险？
- 风险是模糊、近期、主动还是无风险？
- 当前轮能不能继续普通聊天？
- 是否需要安全检查、危机资源、低压微行动？
- 是否要阻断 RAG、普通记忆引用或工具调用？

## risk_classifier

`risk_classifier` 会输出：

- `risk_level`：`L0`、`L1`、`L2`、`L3`
- `risk_domain`
- `immediacy`
- `risk_confidence`
- `protective_signals`
- `risk_reason_codes`
- `risk_response_policy`

它不是只看一个关键词，而是把关键词信号、语义风险、原因码和可选 LLM refine 结合起来。

## control_plane

`control_plane` 是风险之后的控制面。它会输出：

- `route_priority`
- `control_category`
- `clarification_needed`
- `response_contract`
- `memory_policy`
- `rag_policy`
- `tool_gate_mode`

路由优先级很关键：

| 优先级 | 含义 | 后续节点 |
| --- | --- | --- |
| `P0_immediate_safety` | 立即安全优先 | `crisis_response` |
| `P1_red_flag` | 临床红旗 | `clinical_red_flag_response` |
| `P2_support` | 普通支持 | `intent_classifier` |
| `P3_bridge_boundary` | 需要桥接边界 | 多数仍进入受限支持 |
| `P4_system_protection` | 系统保护/边界 | `boundary_response` |

## 为什么是亮点

风控系统不是只在输出后过滤一句“不能回答”。它是全链路控制：

- 影响路由。
- 影响 prompt contract。
- 影响 RAG 是否启用。
- 影响记忆是否可见。
- 影响工具是否可用。
- 影响最终回复按钮。

这让安全策略前置，而不是事后补救。

## 对外讲法

> 项目把心理风险控制做成了图里的早期节点。风险等级不只是一个标签，而会改变后续路由、RAG、记忆引用和回复契约。这样高风险场景不会误入普通咨询或闲聊流程。

## 开发边界

- 风控变更要同步看 RAG gating 和记忆可见性。
- 改 `route_priority` 要同步 `routing.py`。
- 改高风险策略时要检查 `crisis_response` 和 validator。
