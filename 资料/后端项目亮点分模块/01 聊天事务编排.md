# 聊天事务编排

> 返回：[[00 后端项目亮点分模块索引]]
> 相关代码：`backend/app/services/chat_service.py`、`backend/app/services/chat_turn_lifecycle.py`、`backend/app/services/chat_streaming.py`、`backend/app/api/v1/endpoints/chat.py`

## 它解决什么问题

用户点击发送消息，看起来只是一次聊天。但后端真正要保证的是：

- 这条消息属于当前用户和当前会话。
- 重复点击或网络重试不会生成两次不同回复。
- 大模型或 LangGraph 超时要有兜底。
- 用户消息、assistant 消息、trace、风险事件、记忆任务要一起处理。
- 流式接口要能持续给前端反馈，而不是卡住。

这些职责集中在 `chat_service.py`，它是后端“一轮对话事务”的主入口。

## 核心链路

非流式：

```text
API chat.py
  -> process_message_turn
  -> claim_turn
  -> create user message
  -> prepare turn context
  -> graph_runtime.invoke_turn
  -> persist assistant result
  -> complete_turn
  -> return response
```

流式：

```text
API chat.py
  -> process_message_turn_stream
  -> accepted event
  -> graph_update / token / heartbeat events
  -> persist final result
  -> final event
```

## 模块拆分

| 文件 | 职责 |
| --- | --- |
| `chat_service.py` | 一轮消息的业务编排、落库、调用图、兜底、风险事件、记忆任务。 |
| `chat_turn_lifecycle.py` | turn claim、request hash、replay、complete、failed、幂等状态。 |
| `chat_streaming.py` | token chunk、graph update、heartbeat 等流式事件。 |
| `api/v1/endpoints/chat.py` | HTTP/SSE 入口，做鉴权、参数校验和响应包装。 |

## 为什么是亮点

它把“大模型聊天”变成了一个可恢复、可追踪、可持久化的业务事务。很多 agent demo 只关心生成文本，但这个项目关心的是生产级聊天会遇到的问题：

- 重复请求如何处理。
- 图失败时怎么返回。
- trace 怎么保存。
- 记忆写入是否要异步。
- 流式输出中途失败怎么办。

这让后端不是薄薄一层 wrapper，而是有稳定运行边界的对话服务。

## 开发边界

改这个模块时要特别注意：

- 不要绕过 `process_message_turn` / `process_message_turn_stream` 直接写消息。
- 不要破坏 `graph_runtime` 和 `settings` 这类测试 patch 点。
- 修改流式事件时，要同步检查前端 SSE 消费逻辑。
- 修改 turn 状态时，要跑 `tests/test_chat_idempotency.py`。
