# 记忆系统

> 返回：[[00 后端项目亮点分模块索引]]
> 相关代码：`backend/app/services/memory_service.py`、`backend/app/services/memory_scoring.py`、`backend/app/services/memory_job_service.py`、`backend/app/graphs/nodes/memory_nodes.py`

## 它解决什么问题

心理陪伴类 agent 需要连续性，但记忆又非常敏感。记忆系统要解决两组相反的需求：

- 要记得用户的重要偏好、目标、反复出现的困扰、有效支持方式。
- 不能无节制保存敏感内容，不能在高风险场景乱引用记忆，不能让过期记忆误导当前回复。

## 记忆类型

记忆不是一团文本，而是有类型的。常见类型包括：

- `session_summary`
- `goal`
- `state`
- `preference`
- `correction`
- `support_strategy`
- `recurring_trigger`
- `relationship`
- `profile`
- `safety_summary`

这些类型会影响检索优先级、可见性和是否能在高风险场景使用。

## 读记忆链路

聊天前，`chat_service` 会调用：

```text
build_memory_index
retrieve_memories_for_turn_async
build_user_context_pack
build_safety_context_pack
```

记忆检索会综合：

- 用户记忆模式：`off`、`summary_only`、`long_term`
- 当前风险等级
- 记忆类型
- 记忆可见性
- 文本相似度
- 向量分数
- 重要性
- 新鲜度
- access count
- 当前 goal_state 和 control_category

## 写记忆链路

LangGraph 尾部会执行：

```text
summarize_turn
  -> memory_candidate_extract
  -> write_memory / skip_memory
```

`memory_candidate_extract` 会根据当前状态决定是否生成候选记忆。常见策略：

- `failed_no_reply`：不写。
- `memory_mode == off`：不写。
- `skip_sensitive`：不写。
- `crisis_audit_only`：只生成安全摘要类记忆。
- `summary_only`：只保留会话摘要。
- `long_term`：提取结构化长期记忆候选。

业务层还会把真正写入包装成 `PendingMemoryJob`，由后台 worker 处理，避免阻塞聊天主链路。

## 评分与去重

`memory_scoring.py` 负责纯文本评分辅助：

- 清洗文本。
- tokenize。
- term similarity。
- content similarity。
- 判断两个记忆是否值得比较。

`memory_service.py` 在此基础上叠加业务评分：

- 类型 boost。
- 目标相关 boost。
- 冲突/纠偏相关 boost。
- 重要性。
- 新鲜度。
- 访问次数。
- vector score。

## 高风险场景下的记忆控制

当风险等级是 `L2` / `L3` 时，系统不会像普通场景那样自由引用长期记忆，而是：

- 限制可见记忆类型。
- 允许 `internal_safety` 的 `safety_summary`。
- 控制 `user_visible` 记忆是否能进入当前轮。
- 不在危机场景暴露不必要的个人历史。

## 为什么是亮点

这个记忆系统不是简单“把用户说过的话存起来”。它把记忆当作受风险、隐私、类型、时效和对话目标共同约束的上下文资产。

## 对外讲法

> 记忆系统同时支持长期连续性和安全边界。它按记忆类型、可见性和风险等级控制读写，用规则分数加向量分数排序，并通过后台任务写入，避免聊天主链路被记忆持久化拖慢。

## 开发边界

- 修改记忆读取时，要同时看高风险可见性。
- 修改记忆类型时，要同步类型优先级、候选提取和 UI 展示。
- 修改写入策略时，要检查后台 job 和 memory operation 日志。
