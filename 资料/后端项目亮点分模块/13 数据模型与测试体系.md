## 数据模型如何支撑后端能力

后端亮点能成立，是因为数据库模型也比较完整。

核心表：

| 能力 | 表 |
| --- | --- |
| 会话 | `conversation_threads`、`messages` |
| turn 幂等 | `conversation_turns` |
| 图追踪 | `conversation_turn_traces` |
| 风险审计 | `risk_events` |
| 长期记忆 | `user_memories`、`memory_embeddings`、`memory_operations`、`pending_memory_jobs` |
| 知识库 | `knowledge_articles`、`knowledge_sources`、`knowledge_chunks`、`knowledge_gaps` |
| 咨询语料 | `counseling_corpus_sources`、`counseling_example_chunks` |
| 隐私 | `privacy_action_logs` |
| 用户反馈 | `user_feedback` |

这说明项目不是只做运行时逻辑，也有对应的数据沉淀和审计结构。

## 测试体系

后端有不少针对关键链路的测试：

- `test_conversation_control_rag.py`
- `test_conversation_move_policy.py`
- `test_dialogue_prompt_builder.py`
- `test_memory_service.py`
- `test_memory.py`
- `test_chat_idempotency.py`
- `test_graph_runtime_streaming.py`
- `test_response_memory_continuity.py`
- `test_backend_refactor_contract.py`

测试覆盖的重点很贴合项目风险：

- 风控与 RAG 是否正确联动。
- 对话策略是否按用户纠偏变化。
- validator 是否拦住危险或体验差的输出。
- 记忆检索和写入是否符合策略。
- 流式图执行是否能输出正确事件。
- 重构后导入路径和 monkeypatch 点是否稳定。

## 为什么是亮点

测试不是只测工具函数，而是在测心理咨询 agent 最容易出错的行为边界。数据模型也不是简单用户/消息两张表，而是把风险、trace、记忆、知识库、隐私和反馈都纳入了持久化结构。
