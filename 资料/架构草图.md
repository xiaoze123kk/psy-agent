# 一、整体技术架构草图

你们用 LangGraph 很合适，但要注意分层：

Mobile App  
├── 文本聊天  
├── 实时语音聊天  
├── 情绪记录  
├── 测试中心  
├── 心理知识库  
└── 我的 / 记忆 / 隐私 / SOS  
  
Backend API  
├── Auth Service  
├── User/Profile Service  
├── Chat Service  
├── Voice Realtime Service  
├── Test Service  
├── Knowledge RAG Service  
├── Memory Service  
├── Safety Service  
└── Analytics/Feedback Service  
  
LangGraph Agent Layer  
├── conversation_graph  
├── safety_graph  
├── knowledge_graph  
├── test_interpretation_graph  
├── anime_match_graph  
└── memory_update_graph  
  
Data Layer  
├── PostgreSQL  
├── Redis  
├── pgvector / 向量库  
├── Object Storage，可选  
└── LangGraph Checkpointer / Store

LangGraph 的核心概念是 **State、Nodes、Edges**：State 是共享状态，Nodes 执行业务逻辑，Edges 决定下一步路由；这正好适合你们这种“先风险判断，再分流到陪伴/知识/测试/危机”的 agent。

建议你们用：

App：Flutter 或 React Native  
Backend：Python FastAPI  
Agent：LangGraph Python  
DB：PostgreSQL + pgvector  
缓存/队列：Redis  
实时语音：WebSocket / WebRTC  
对象存储：MinIO / S3，可选

LangGraph 的 checkpoint 适合保存单个会话线程的图状态，thread_id 可以理解成一段对话的持久游标；而跨会话长期记忆更适合用 Store 或你们自己的 user_memories 表来维护。官方文档也区分了 thread checkpoint 和跨 thread 的 Store memory。

### 实时语音的产品策略

实时语音不要一开始做得太复杂，建议 MVP 做成：

用户音频流  
-> 实时 STT  
-> 分段转写  
-> 语音活动检测 VAD  
-> 话轮结束  
-> 把完整用户话轮送入 LangGraph  
-> LLM 生成文本  
-> TTS 合成语音  
-> 流式播放

LangGraph 负责“完整话轮”的心理陪伴编排，不建议每个音频帧都进 LangGraph。实时音频链路对低延迟要求更高，应该放在 WebSocket/WebRTC 服务里；LangGraph 用于话轮级推理、风险判断、记忆检索、回复生成和会话摘要。

### 首批动漫作品池建议

截至 2026-04-24，近两年热度和奖项曝光较高的作品可以先从下面选 8–10 部。2025 年 Crunchyroll Anime Awards 的年度动画提名包括《DAN DA DAN》《迷宫饭》《葬送的芙莉莲》《怪兽 8 号》《我独自升级》《药屋少女的呢喃》；2026 年提名中又出现了《DAN DA DAN Season 2》《药屋少女的呢喃 Season 2》《Gachiakuta》《我的英雄学院 FINAL SEASON》《光死去的夏天》《章鱼哔的原罪》等。

MVP 我建议先做这些：

|优先级|作品|原因|
|---|---|---|
|P0|葬送的芙莉莲|角色性格维度丰富，适合心理解释|
|P0|药屋少女的呢喃|主角特征鲜明，适合人格映射|
|P0|DAN DA DAN / 胆大党|年轻用户热度高，角色反差大|
|P0|我独自升级|成长线强，适合“压力应对/逆境成长”|
|P0|怪兽 8 号|普通人转变、责任感主题明显|
|P1|迷宫饭|群像角色多，适合多类型匹配|
|P1|我推的孩子|情绪、身份、表演人格主题强|
|P1|咒术回战|传播度高，但角色心理解释要避免过暗|
|P1|SAKAMOTO DAYS|新作热度潜力|
|P1|光死去的夏天|氛围强，但青少年模式要谨慎处理|

注意：开源项目里不要直接放官方角色图片、截图、台词大段引用。先做 **文字结果 + 原创抽象头像 + 角色人格向量**。

# 三、LangGraph 核心设计

## 3.1 图拆分建议

不要把所有功能塞进一个巨大的 graph。建议拆成 6 个 graph：

conversation_graph          核心陪伴对话  
safety_graph                风险识别与危机分流  
knowledge_graph             心理知识 RAG  
test_graph                  测试计分与解释  
anime_match_graph           动漫角色匹配  
memory_graph                会话摘要与长期记忆写入

实际调用时，可以由主 graph 编排子 graph。

main_agent_graph  
├── safety_graph  
├── conversation_graph  
├── knowledge_graph  
├── test_graph  
├── anime_match_graph  
└── memory_graph

## 3.5 LangGraph 路由伪代码

def route_by_risk(state: AgentState) -> str:  
    if state["risk_level"] in ["L2", "L3"]:  
        return "crisis_response"  
    return "intent_classifier"  
  
  
def route_by_intent(state: AgentState) -> str:  
    intent = state["intent"]  
  
    if intent == "knowledge_qa":  
        return "knowledge_retrieve"  
    if intent == "test_interpretation":  
        return "test_context"  
    if intent == "anime_match":  
        return "anime_context"  
    if intent == "soothe":  
        return "soothing_response"  
    if intent == "light_counseling":  
        return "counseling_response"  
  
    return "companion_response"

LangGraph 里可以用 conditional edges 做这种分流；如果某个节点既要更新状态又要决定跳转，则可以返回 `Command(update=..., goto=...)`。官方文档也建议：只需要路由时用 conditional edges；需要同时更新状态和控制流时用 Command。

## 3.6 Checkpointer 与 thread_id 设计

每一个对话线程对应一个 LangGraph thread：

conversation_threads.langgraph_thread_id

调用 graph 时：

config = {  
    "configurable": {  
        "thread_id": thread.langgraph_thread_id,  
        "user_id": user_id  
    }  
}

LangGraph 的持久化层会把 graph state 按 thread 保存 checkpoint；这可以支持会话续接、debug、回放、故障恢复。

---

## 3.7 流式输出设计

文本聊天建议用：

SSE 或 WebSocket  
stream_mode = ["messages", "updates"]

其中：

|stream mode|用途|
|---|---|
|messages|流式返回 LLM token|
|updates|返回节点执行状态，例如正在检索知识、正在生成回复|
|values|调试时看完整状态|
|debug|开发环境调试|

LangGraph 官方文档说明，`messages` 可以流式返回 LLM token，`updates` 可以返回每个节点后的 state update，适合聊天应用展示“正在理解 / 正在检索 / 正在回复”。

## 3.8 实时语音与 LangGraph 的关系

推荐结构：

App 麦克风  
  ↓  
Voice Gateway WebSocket  
  ↓  
VAD 判断话轮  
  ↓  
Streaming STT  
  ↓  
完整用户话轮 transcript  
  ↓  
LangGraph conversation_graph  
  ↓  
assistant_text 流式返回  
  ↓  
Streaming TTS  
  ↓  
App 播放语音

### 不建议

每 100ms 音频帧 -> LangGraph

这样会导致延迟高、checkpoint 过多、状态污染严重。

### 建议

每个完整话轮 -> LangGraph

例如用户说完一句：

> “我今天真的很难受，我感觉没人理解我。”

再进入 graph。

# 八、你们现在最该先实现的最小闭环

最小闭环不要一开始做全部页面。建议先做这个：

注册登录  
→ 年龄确认  
→ 新手引导  
→ 文本聊天  
→ 风险识别  
→ LangGraph 回复  
→ 会话摘要  
→ 记忆写入  
→ 下次回来继续聊

这个闭环跑通后，再加：

情绪记录  
→ 16 型人格探索  
→ 动漫角色匹配  
→ 实时语音

核心判断标准不是功能数量，而是：

用户第二次回来时，AI 能不能自然地接住：  
“我记得你上次说最近晚上容易焦虑，今天也是这种感觉吗？”

这才是你们这个心理陪伴 agent 的核心体验。
